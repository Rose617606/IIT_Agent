# 我用 73 份法规喂了一个 AI（一）：你以为缺的是算法，其实缺的是干净数据

> 年终奖怎么选计税方式？房租和房贷能同时扣吗？——你可能需要一个能读懂税法的 AI。

---

## 钩子：年终奖发下来，财务让你"二选一"，你懵了

小张，30 岁，月薪 2 万，在上海租房，父母在老家。

年底公司发了 8 万年终奖，财务发来一条消息："请选择年终奖计税方式——单独计税，还是并入综合所得？"

他盯着屏幕看了三分钟，打开了百度。

搜索结果是：《财政部 税务总局关于延续实施全年一次性奖金个人所得税政策的公告》《个人所得税法》第六条、国发〔2018〕41号、2023 年第 30 号公告……

**每个字都认识，连起来完全看不懂。**

他最后凭感觉选了"单独计税"。三个月后汇算清缴，同事退税 3000，他补税 2000。

就是那一刻，我决定做 IITAgent——一个能读懂中国个税法规、帮你做决策的 AI 智能体。

## 痛点：你不是不懂税，你是被法条淹没了

小张不是个例。每年 3 到 6 月个税汇算清缴，数千万人面对同一个困境：

- "子女教育和婴幼儿照护能叠加吗？"
- "租房和房贷利息到底能不能同时扣？"
- "我该不该把年终奖单独计税？"
- "赡养老人怎么证明？兄弟姐妹怎么分摊？"

答案都在税法里。但税法长这样：

> 居民个人取得综合所得，按纳税年度合并计算个人所得税；非居民个人取得综合所得，按月或者按次分项计算个人所得税。综合所得，适用百分之三至百分之四十五的超额累进税率。

**你需要的是一个翻译官，不是一个搜索引擎。**

所以我动手做了一个。不是"把你的问题丢给 ChatGPT"那种——那种会编造不存在的法规条款。我要的是一个真正读过税法、能准确引用条文、信息不全时会反问你的 AI 智能体。

但说起来容易。第一步就撞墙了：**去哪找"税法"？**

## 认知颠覆：你以为的"知识库构建"，其实是一条信息流水线

如果你把 RAG 理解成：

> 用户问题 → 检索几段文本 → 拼到 Prompt 里 → 让模型回答

那你很容易陷入一个误区：**只关注检索和生成，却忽略了检索之前的数据准备阶段**。

在真实工程里，RAG 更像一条完整的信息流水线：

```
离线索引阶段                        在线检索阶段
───────────                        ───────────
文档加载                            查询预处理
  ↓                                   ↓
内容抽取（保留标题+表格）            召回（Dense + Sparse）
  ↓                                   ↓
脏数据清洗                          元数据过滤
  ↓                                   ↓
结构恢复                            RRF 融合
  ↓                                   ↓
Chunk 切分（语义边界优先）          Reranker 精排
  ↓                                   ↓
Metadata 增强（LLM提取文号/日期）    上下文拼接 → LLM 生成
  ↓
Embedding 生成（BGE-M3 双路向量）
  ↓
向量索引构建（Chroma + Sparse 索引）
```

**离线阶段解决的是"知识如何被表示"，在线阶段解决的是"知识如何被找到和使用"。**

很多人 RAG 翻车，不是因为模型太弱，而是因为答案在进入索引前，就已经被错误地表示了。

## 解决方案：三层递进的文档预处理实战

### 第 1 层：结构化提取 — 别把 Word 当纯文本处理

第一版代码很简单：

```python
def extract_text(docx_path):
    doc = Document(docx_path)
    return "\n".join([p.text for p in doc.paragraphs])
```

跑完之后发现三个致命问题：

**问题 1：标题层级全丢了。** 中国政府文档几乎不用 Word 的标题样式（"Heading 1"、"Heading 2"），而是纯靠**加粗**和**"第X条""一、"** 这样的文本模式来组织层级。如果全转成纯文本再按固定长度切，极有可能把"第一条"（定义）和"第二条"（适用范围）切进同一个 chunk。

**问题 2：表格全丢了。** 很多法规有税率表、扣除标准表。`p.text` 只读段落文字，表格里的内容直接蒸发。税率表是最需要被检索到的信息之一。

**问题 3：噪声污染。** 页眉页脚、版权声明、页码、多余空白符混进正文。Embedding 模型不是魔法——如果一段文本里混入了噪声，它生成的向量本质上就是在为"噪声 + 有效语义"的混合体建模。

修复方案：

```python
def extract_text(docx_path):
    doc = Document(docx_path)
    body = []
    
    # 按文档内元素顺序遍历（段落与表格交替）
    for child in doc.element.body:
        if is_paragraph(child):
            para = get_paragraph(child)
            level = detect_heading_level(para)  # 样式 + 加粗 + 文本模式
            if level:
                body.append("#" * level + " " + para.text)
            else:
                body.append(para.text)
        elif is_table(child):
            body.append(table_to_markdown(child))  # 保留行列关系
    
    return clean_text("\n\n".join(body))  # 空白归一、去页眉页脚
```

标题检测三层判断：
1. **样式名**（"Heading 1" / "标题 1" — 有则用）
2. **加粗 + 短段落**（政府公文最常用：标题纯靠 Ctrl+B）
3. **内容模式**（"第X条"→H2，"一、"→H3）

效果对比：

| 指标 | 修复前 | 修复后 |
|------|--------|--------|
| 标题层级 | 0 | 151 H2 + 557 H3 |
| 表格内容 | 全部丢失 | 保留 |
| 页眉页脚噪声 | 混杂 | 清洗 |

### 第 2 层：元数据提取 — 让检索可过滤、可溯源

只存 `chunk_text + embedding`，能跑，但不够工程化。

我们需要的元数据：

```yaml
---
title: 个人所得税专项附加扣除暂行办法
source: 国发〔2018〕41号
effective_date: 2019-01-01
status: active
---
```

怎么填？两种极端方案：

- **正则匹配**：维护 103 份文件的正则规则，文号格式有 5 种以上变体，日期表达有 10 种以上，维护性灾难。
- **LLM 逐个提取**：103 次 API 调用，准确率高但有点浪费。

实际用了**混合方案**：**LLM 批量提取**。每批 20 个文件的前 800 字，6 次 API 调用，DeepSeek 成本不到 $0.02。

关键提示词约束：

```
- source 必须是标准文号（如 "国发〔2018〕41号"）。
  没有文号的文件，必须填空字符串 ""，
  绝对不要填发文机关名、文件名的一部分
```

为什么要这么严格？因为 LLM 有个坏习惯——看到"国家税务总局公告"这样的发文机关名，会把它当成文号填进去。必须明确禁止。

最终结果：**48/73 文号命中，17/73 生效日期命中**。没有命中的那些是问答、解读、案例类文档，本来就没有文号——LLM 正确识别了它们，返回了空字符串。这就是"准确的不填"比"错误地乱填"重要。

### 第 3 层：Chunk 切分 — 不设硬上限，语义边界优先

很多教程让你直接写：

```python
splitter = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=100)
chunks = splitter.split_text(text)
```

这在税务法规场景下会出问题：一个条款可能 600 字，硬切会把"扣除标准"和"扣除方式"拆成两个 chunk。检索召回时，"看起来相关"但"回答总是漂"。

我们的策略：**语义边界优先，字符数仅作软约束**。

```python
# 优先按文档结构切分（段落 → 分句 → 逗号），最后才强制字符限制
separators = ["\n\n", "\n", "；", "。", ";", ".", "，", ",", " "]
max_chunk_size = 500   # 软上限，非硬限制
chunk_overlap = 80     # ~16%
```

设计文档里写明了：**不设硬性字符数上限**。能按完整段落独立成段的，绝不强制切碎。只有超长段落才用 RecursiveCharacterTextSplitter 兜底。

## 技术栈全景

| 层 | 选型 | 理由 |
|----|------|------|
| **Embedding** | BGE-M3 (568M) | 唯一同时输出 Dense + Sparse 双路向量的开源模型，一个模型替代"向量检索 + BM25" |
| **Reranker** | BGE-Reranker-v2-m3 | 与 BGE-M3 同生态，Cross-Encoding 精排 |
| **向量库** | Chroma | 轻量、本地、LangChain 原生支持 |
| **Agent 框架** | LangGraph | `interrupt()` 机制实现反问等待的确定性暂停 |
| **LLM** | DeepSeek | 中文友好、性价比高 |
| **计算** | Python Decimal | 金额字段禁止 float |

为什么选 BGE-M3 而不是精度更高的 Qwen3-Emb？

BGE-M3 的 Dense + Sparse 双路输出**一个模型搞定混合检索**。税务 RAG 必须做混合召回——既有语义覆盖（"月薪两万"匹配到"工资薪金所得"），又有关键词精确匹配（"专项附加扣除""子女教育"等固定术语）。Qwen3 语义精度更高但只输出 Dense 向量，需要额外引入 BM25。

这个结论来自两轮 GitHub 调研，对比了 7 个开源税务/法律 AI 项目后的决策。
