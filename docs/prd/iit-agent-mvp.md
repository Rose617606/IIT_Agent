## 1. 产品概述

### 1.1 项目定位

**IITAgent（个税智能体）** 是一个精通中国个人所得税法规的 AI 智能体，为普通纳税人提供**税前筹划**和**合规解读**服务。

不同于"帮你填写个税 APP"的申报工具，IITAgent 的定位是**决策顾问**：核心价值是"事前规划"和"可信解释"——在你还未操作之前，帮你比较不同计税方案、讲清楚每条规则的含义。

### 1.2 三大核心能力

| 能力 | 描述 | 示例 |
|------|------|------|
| **翻译官** | 法条转白话，让普通人听懂 | "累计预扣法" → "每个月公司会把你之前的工资累加起来算税，年底多退少补" |
| **咨询者** | 反问补全信息，不乱猜 | 用户问"我该不该单独计税年终奖？"，Agent 反问"你全年综合收入大概多少？每月有房租支出吗？" |
| **结果比对器** | 精准计算 + 方案对比 | 年终奖单独计税 vs 并入综合所得，输出税后到手差额和推荐方案 |

## 2. 目标用户与场景

### 2.1 目标用户

- **首发用户**：中国境内工薪阶层（有综合所得、可能涉及年终奖、专项附加扣除）
- **次级用户**（V1.1+）：HR、财务人员（批量计算、企业角色）

### 2.2 MVP 首发场景

| 优先级 | 场景 | 用户需求 |
|--------|------|---------|
| P0 | **年终奖单独计税 vs 并入综合所得对比** | 财务发年终奖前，想知道哪种方式到手更多 |
| P0 | **专项附加扣除解读** | 租房/买房/有孩子/赡养老人的用户，想确认自己能扣多少 |
| P1 | **综合所得汇算清缴规则解读** | 每年 3-6 月，用户想知道自己需不需要补税 |
| P1 | **多扣除项叠加计算** | 同时有房租、赡养老人等多项扣除，想算综合优惠 |

### 2.3 典型用户故事

> 小张，30岁，月薪 20000，在上海租房，父母在老家（他是独生子女）。公司发了 8 万年终奖，财务让他选计税方式。
>
> 他问 IITAgent："我年终奖 8 万，月薪 2 万，租房加赡养老人，哪种计税方式省税？"
>
> Agent 反问："你的社保公积金每月大概扣多少？有没有子女教育或房贷利息？" → 小张补充 → Agent 输出两种方案的详细对比，并解释为什么。

## 3. 功能边界

### 3.1 MVP 包含

- 综合所得年度汇算税额计算
- 年终奖单独计税 → 税额计算
- 年终奖并入综合所得 → 税额计算
- 两种年终奖方案的**对比输出**
- 6 项专项附加扣除的规则查询 + 白话解释
- 信息不全时的**反问等待**
- 所有回答**引用法规出处** + 免责声明

### 3.2 MVP 明确不做

- 经营所得、财产转让、股息红利等其他 8 类所得
- 外籍人士特殊规则
- 个税申报表单生成/导出
- 真实用户认证系统
- 联网查税法（全部依赖知识库离线数据）

### 3.3 V1.1 计划（Phase 6）

- 对话持久化（SQLChatMessageHistory）
- 政策更新自动监控
- 多轮情景模拟（what-if 分析）
- 合规熔断机制

## 4. 验收标准 (Acceptance Criteria)

### 4.1 检索质量

| ID | 场景 (Given) | 行为 (When) | 预期 (Then) |
|----|-------------|------------|-------------|
| AC-01 | 用户问"子女教育怎么扣" | Agent 检索知识库 | 返回包含"每个子女每月2000元"的正确条文，引用来源为国发〔2018〕41号或国发〔2023〕13号 |
| AC-02 | 用户问"房租和房贷能同时扣吗" | Agent 检索 | 返回明确结论"不能同时享受"，引用法规原文 |
| AC-03 | 用户问"年终奖怎么算税" | Agent 检索 | 返回两种计税方式的计算规则和税率表 |

### 4.2 计算准确性

| ID | 场景 | 预期 |
|----|------|------|
| AC-04 | 月薪 20000，年终奖 80000，无专项附加扣除 | 两种方案计算误差 ≤ 1元（与官方个税 APP 对比） |
| AC-05 | 月薪 15800，年终奖 36000，有住房租金扣除 | 两种方案计算误差 ≤ 1元 |
| AC-06 | 边界值：年终奖恰好位于税率跳档临界点（如 36000、144000） | 计算正确，不会误用更高税率 |

### 4.3 交互质量

| ID | 场景 | 预期 |
|----|------|------|
| AC-07 | 用户问"哪种方式省税"，未提供收入信息 | Agent 反问缺失参数（至少问月薪、年终奖金额），不乱猜 |
| AC-08 | 用户提供完整信息后 | Agent 在 3 秒内输出对比结果 |
| AC-09 | 所有回答末尾 | 包含法规出处引用 + 免责声明"本结果仅供参考，具体以税务机关核定为准" |

### 4.4 整体目标

- 20 个典型测试问题准确率 > **80%**（Phase 5 评测）
- 无幻觉 — 不得编造不存在的法规条款
- 检索 + 生成延迟 < **3 秒**

## 5. 非功能需求

### 5.1 性能

- 向量检索延迟 < 1 秒
- BGE-M3 编码延迟 < 200ms（CPU 模式）
- 端到端回答延迟 < 3 秒（不含 LLM 推理时间）

### 5.2 合规

- 所有计算引用明确法规条款号
- 输出末尾强制追加免责声明
- 不得提供任何形式的"逃税建议"

### 5.3 技术约束

- 金额字段必须使用 `Decimal`，禁止 `float`
- 知识库数据必须离线可用（不依赖外网查税法）
- 本地运行，不将用户财务数据上传第三方

## 6. 依赖与风险

| 依赖/风险 | 类型 | 说明 | 缓解措施 |
|-----------|------|------|---------|
| 政策文件质量 | 依赖 | 知识库依赖手动整理的 Markdown 政策文档 | 挑选权威来源，标注生效日期和失效标记 |
| 税法变动 | 风险 | 扣除标准每年可能调整 | `is_expired` 标记 + 紧急覆盖规则 |
| BGE-M3 上下文限制 | 风险 | 8K token，长政策全文可能截断 | 切片控制在 300 字以内；过长的政策拆分为多份文档 |
| LLM 幻觉 | 风险 | 模型可能编造法规 | RAG 强制检索 + 引用溯源 + 三重校验 |
| 无 GPU | 约束 | BGE-M3 在 CPU 上运行较慢 | 接受每请求 200ms 延迟；一次编码后可缓存 |
